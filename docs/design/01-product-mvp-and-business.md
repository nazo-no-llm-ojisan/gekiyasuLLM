## 1. エグゼクティブサマリー

**gekiyasuLLM** は、分散した LLM API の価格・無料枠・能力・可用性・データ利用条件を継続収集し、**署名可能なルーティングフィード**として配信する中央情報サービスと、各利用者の PC 上で **OpenAI/Anthropic 互換 API** を提供し、ポリシーに従ってエンドポイントを選ぶ **ローカル OSS プロキシ** の二層構成である。

中核価値は「**コストと能力と信頼ポリシーの範囲で、自分の鍵でルーティングする**」ことであり、リクエストを中央が中継・課金するタイプの API アグリゲータとは信頼境界が異なる。類似領域には自前 LLM ゲートウェイ（例: LiteLLM, Portkey, Helicone）や、ベンチ・価格比較サイト（例: Artificial Analysis）がある。本構想は **無料枠・キャンペーン・互換性・ローカルポリシー付きルーティング** に焦点を置く。

**MVP** は個人が当日使える最小セットに絞る: OpenAI 互換ローカル Proxy、静的（後に署名付き）フィード、価格・能力制約によるルーティング、retry/fallback/circuit breaker、ローカル統計、秘密情報ポリシー、CLI。中央 DB・認証・広告・telemetry・Web UI は後段。

**推奨スタック**: ローカル Proxy は **TypeScript (Node.js)**（後に Go 単一バイナリ化を検討）、ソースは **GitHub 公開リポジトリ**、フィード生成に **GitHub Actions**、サイトは **Cloudflare Pages**、将来 API/集計は **Cloudflare Workers (+ D1 または Neon)**。

**最大リスク**: (1) フィード品質と価格陳腐化、(2) 信頼度の低い第三者エンドポイントへ誤って機微データを送ること、(3) 可用性プローブが規約違反・過負荷と見なされること、(4) 広告・紹介と編集評価の混同による信頼失墜。

## 2. 問題設定

### 2.1 解決する問題

LLM API の選択情報（価格、キャッシュ価格、定額、無料枠、CC 要否、コンテキスト長、互換性、tool/stream/vision、可用性、TTFT、レート制限、運営主体、学習利用条件、キャンペーン終了リスク）が **公式・第三者・SNS・個人メモに分散**し、比較コストが高い。さらに情報が **日次で陳腐化**する。

利用者は IDE / コーディングエージェントから、「条件を満たす範囲でコストを抑えつつ動くモデル」へ繋ぎたいが、

- 手動切替は面倒
- 中央中継は鍵とプロンプトの信頼境界が曖昧になりやすい
- 既存ゲートウェイは企業向け制御や課金アグリゲーションが中心で、**無料枠・キャンペーンを一等市民にした個人向けルーティング**が手薄になりやすい

### 2.2 想定利用者

| セグメント | ニーズ | 信頼度要件 |
|---|---|---|
| 個人開発者 / 学生 | コストを抑えてコーディングエージェントを回す | 低〜中（公開コード中心） |
| インディー / 副業開発者 | コスト上限、fallback、日次変化通知 | 中 |
| 企業個人利用 | 社内ポリシー、private code の送信制限 | 高 |
| 将来: 法人監視 | SLA 監視、監査、allowlist 管理 | 高 |

### 2.3 中核的な価値

1. **情報の一元化と検証可能性**（価格・能力・観測・スポンサー関係の分離）
2. **ローカル実行のルーティング**（鍵・プロンプトは原則ローカルから直接プロバイダへ）
3. **ポリシー駆動**（コスト、能力、private、信頼度、allowlist）
4. **運用の自動化**（fallback / circuit breaker / 予算 / 統計）
5. **透明性**（スポンサー・affiliate を明示、編集評価と分離）

### 2.4 リポジトリとフェーズのスコープ

**公開前提で進めてよいこと**

- 設計ドキュメントの更新（読まれても問題ない表現・免責を維持）
- **個人向け MVP 実装**（Phase 1〜3: ローカル Proxy、静的フィード、health / ローカル統計、CLI）
- 公開用サンプルフィード（秘密・個人 API キー・署名秘密鍵を含まない）
- ローカルでの動作確認（API キーは環境変数等。リポジトリに入れない）

**後段・メンテナ承認が必要なこと**

- 中央でのユーザーリクエスト中継
- 本番インフラの有料契約、商用課金の本番運用
- 観測値を「保証」や「公式認定」のように見せる表記
- 法務レビュー済みの確定表示文言（注意点の記載までは可）
- Phase 6 以降の月額・広告ランタイムの本実装（設計は可、実装は方針合意後）

## 3. サービス企画

### 3.1 製品構成

| 仮称 | 役割 |
|---|---|
| **gekiyasuLLM.com** | 情報サイト + フィード配信 + 日次ヘッドライン +（将来）有料購読 |
| **gekiyasuLLMProxy** | ローカル OSS Proxy（ルーティング実行主体） |

### 3.2 競合・代替手段

| 類型 | 例 | 本構想との差分 |
|---|---|---|
| マネージド API アグリゲータ | 例: OpenRouter 等 | 中央がリクエスト・課金を中継。鍵・データ境界が異なる |
| 自前 LLM Gateway | 例: LiteLLM, Portkey, Helicone | 企業向け制御・観測が主。無料枠・キャンペーンの継続収集とローカルポリシー連携は主目的ではないことが多い |
| ベンチ・比較サイト | 例: Artificial Analysis 等 | 価格・速度の公開比較は強いが、ローカルポリシー付き自動ルーティングは通常持たない |
| 手動比較 | 各公式 docs / ブログ | 情報は正確になりやすいが比較コストが高い |
| IDE 標準プロバイダ切替 | Cursor / Continue 等の設定 | 静的で、可用性・日次価格変化に弱い |

**差別化の狙い**: 「中継しない」「無料枠・キャンペーンを一等市民にする」「信頼度とデータ条件をルーティング制約にする」「編集と広告をスキーマで分離する」。

### 3.3 無料 OSS と有料の境界

| 無料 OSS | 有料（将来） |
|---|---|
| Proxy 本体 | 高頻度フィード（例: 1h 更新） |
| 公開静的フィード（日次〜数時間） | アラート優先配信 |
| 基本 CLI / 設定スキーマ | 履歴 API・差分 API |
| ローカル統計 | 法人向け監視ダッシュボード |
| コミュニティ貢献の provider 定義 | スポンサー枠（明示ラベル付き） |
| | affiliate 経由の紹介（明示） |

**原則**: 個人がオフラインで動くコアは OSS のまま。中央の「鮮度・網羅・検証工数」を有料化する。

### 3.4 段階的発展

1. **公開リポジトリ上の設計 + 個人向け Proxy 骨格**
2. **利用可能な MVP**（ルーティング・フィード・fallback）
3. **署名付きフィードと貢献ガイド**（OSS として外から使いやすい状態）
4. **情報サービス**（サイト・RSS・日次）
5. **低額購読**（約 $1/月 想定のプレミアムフィード。価格は未確定）
6. **広告・affiliate**（明示フィールド）
7. **法人向け**（監査・allowlist・監視。任意）

### 3.5 収益源（想定）

- 月額情報配信（~$1/月、将来）
- スポンサー枠（フィード内 `sponsored: true`、UI で区別）
- affiliate（明示 + クリック経由）
- 法人監視・専用 probe 地域
- 寄付 / GitHub Sponsors（OSS 維持）

詳細は §17 も参照。

### 3.6 広告と編集評価の分離

- **編集データ**: `source: "official" | "observed" | "estimated" | "community"`
- **商業関係**: `relationships.sponsored`, `relationships.affiliate` を独立フィールド
- ランキング既定ソートは **スポンサー非依存**（コスト・能力・可用性のみ）
- スポンサー枠は別セクション / 別フラグでのみ表示
- 「おすすめ」文言は編集基準を公開し、スポンサー優先を禁止

### 3.7 法務・表示・ステルスマーケ回避

- 価格・無料枠は **取得日時・情報源 URL** 必須
- 「最安」「おすすめ」は時点・条件付きと明記し、保証と誤読されないようにする
- アフィリエイトリンクは `rel` / 表記で明示
- ステルスマーケティング（日本の景表法・ステマ規制意識）を避け、**有償関係は常に開示**
- プロバイダ利用規約に反する probe / スクレイピングは行わない
- 本サービスは **LLM 出力の正確性・可用性を保証しない** 免責を表示
- プライバシー: telemetry は opt-in、プロンプト本文は収集しない

### 3.8 名称・ドメイン評価

| 名称 | 評価 |
|---|---|
| **gekiyasuLLM** / gekiyasuLLM.com | 日本語市場で「激安」が直感的。英語圏では意味不明。ローマ字+LLM は SEO でニッチ |
| gekiyasuLLMProxy | 役割は明確。長い |

**代替案（候補）**:

| 案 | ニュアンス |
|---|---|
| **thriftLLM** / thriftllm.dev | 英語で「倹約」。技術者向け |
| **bargainRouter** | ルーターであることが明確 |
| **cheaplane** | 短い。レーン=経路 |
| **modelfrugal** | 技術寄り |
| **yenpath** | 日本発を想起。国際展開時は再検討 |
| **route.cheap** | ドメイン戦略向き（取得可否要確認） |

**推奨**: 国内個人向けは **gekiyasuLLM** を維持可。国際 OSS 公開時は **英語副ブランド**（例: thriftLLM）を併記。ドメインは `.com` 優先、取得不可なら `.dev` / `.io` + 将来転送。

## 4. MVP

### 4.1 MVP ゴール

「IDE の API base URL を `http://127.0.0.1:8787/v1` に向け、自分の API キーとローカル設定だけで、**能力制約を満たす候補のうちコスト優先で**送り、落ちたら次へ fallback する。」

本リポジトリは **GitHub 公開前提**であり、企画専用に凍結しない。上記ゴールに向かう実装を、設計書と同じ公開リポに置いてよい。

### 4.2 MVP に含める

| 項目 | 内容 |
|---|---|
| OpenAI 互換 Proxy | `/v1/chat/completions`, `/v1/models`（最低限） |
| 静的フィード | ローカルファイル or URL の JSON |
| provider / model 定義 | 価格、context、capabilities、compat |
| ルーティング | hard constraints + コスト昇順 |
| health probe | 任意エンドポイントの軽量 probe（オプトイン） |
| retry / fallback / circuit breaker | 基本実装 |
| ローカル利用統計 | SQLite または JSONL |
| 秘密情報ポリシー | 簡易 secret redaction + allowlist |
| CLI | `serve`, `route dry-run`, `stats`, `feed pull` |
| 設定 | YAML/TOML、hot reload は望ましいが file watch で可 |

### 4.3 MVP に含めない

- Anthropic `/v1/messages`（Phase 1.5 以降）
- OpenAI `/v1/responses`（後続）
- 中央 DB・ユーザー認証
- telemetry 送信
- Web UI（統計は CLI）
- 署名検証の必須化（個人用は任意、公開フィード段階で必須）
- rtk 必須統合（optional adapter のみ設計）
- 広告・affiliate ランタイム
- bandit 学習ルーティング
- multimodal 完全対応
- 実課金との自動突合
- Docker 必須配布（任意は可）
- 日次ニュースレター本番

### 4.4 MVP 完了条件

1. Cursor / Continue / 任意 OpenAI SDK から 1 リクエスト成功
2. 無料候補失敗時に有料候補へ自動 fallback
3. private モードで非許可 provider が選ばれない
4. 推定コストと成功/失敗がローカルに残る
5. フィードを差し替えるだけで候補集合が変わる

## 17. 収益化案

| 段階 | 手段 | 価格感 | 前提 |
|---|---|---|---|
| OSS | Sponsors / 寄付 | 任意 | 信頼蓄積 |
| 情報 | プレミアムフィード | ~$1/月 | 更新頻度・網羅 |
| 広告 | スポンサー枠 | 枠売り | **編集非干渉** |
| 紹介 | affiliate | 成果報酬 | 明示リンク |
| 法人 | 監視・専用地域 probe | $20–200/月 帯 | SLA 説明 |

**反パターン**: スポンサーを、コスト順などの既定ランキングに混ぜる。

## 推奨案（3 段階）

### A. 個人利用向け最小構成

| 項目 | 内容 |
|---|---|
| **構成** | TS Proxy + ローカル feed JSON + SQLite/JSONL 統計 + CLI |
| **主要サービス** | なし（任意で Git バックアップ） |
| **月額概算** | **$0** |
| **実装工数** | **1.5–3 人週**（互換・stream で振れ幅） |
| **運用負荷** | 低（自分の feed を週次メンテ） |
| **最大リスク** | 価格情報の陳腐化、互換バグ |

### B. 公開 OSS 向け構成

| 項目 | 内容 |
|---|---|
| **構成** | A + 公開 schema + 署名フィード + CF Pages サイト + Actions 日次 + RSS |
| **主要サービス** | GitHub, Cloudflare Pages/DNS,（任意）UptimeRobot, CF Analytics |
| **月額概算** | **$0 + ドメイン年額**（目安 $10–20/年） |
| **実装工数** | **+3–6 人週**（ドキュメント・CI・署名・サンプル） |
| **運用負荷** | 中（フィード更新、Issue 対応） |
| **最大リスク** | 誤った価格・可用性情報による信頼失墜、サポート過負荷 |

### C. 商用サービス向け構成

| 項目 | 内容 |
|---|---|
| **構成** | B + 有料フィード API + Stripe + Resend + Workers/D1 or Neon + 中央 probe + 開示付きスポンサー |
| **主要サービス** | Cloudflare Workers Paid, Neon/D1, Stripe, Resend, Sentry |
| **月額概算** | **$10–50**（初期。probe 課金は別途 LLM API 費） |
| **実装工数** | **+6–12 人週**（課金・法務表示・probe 設計） |
| **運用負荷** | 中〜高（日次品質、課金、開示監査） |
| **最大リスク** | 不当表示・利益相反、probe 規約違反、LLM API 費用肥大 |
